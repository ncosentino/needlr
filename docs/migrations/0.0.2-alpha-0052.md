# Migrating to 0.0.2-alpha.52 — Tool Argument Boundary

This guide walks through migrating a codebase that worked around the source-generator bug fixed in [0.0.2-alpha.52](https://github.com/ncosentino/needlr/blob/main/CHANGELOG.md). The canonical example is the BrandGhost case where 9 production `[AgentFunction]` tools were rewritten to use `JsonElement` parameters and forward to private `string` overloads through a hand-written normalizer.

## Who this is for

You hit one or more of these symptoms in 0.0.2-alpha.51 or earlier:

- **The production crash:** `System.InvalidOperationException: The requested operation requires an element of type 'String', but the target element has type 'Array'` thrown from a generated `AIFunction` wrapper, before your tool body ran.
- **The workaround:** a hand-written helper (e.g. `JsonArgumentNormalization`) that converts `JsonElement` back to a string before passing it to your tool body.
- **The defensive policy:** an architecture / reflection test that fails the build if any `[AgentFunction]` parameter named `*Json` / `*_json` (or whose `[Description]` mentions "JSON array" / "JSON object") is typed as `string`.
- **The duplicated code:** every JSON-shaped tool refactored to a public `JsonElement` parameter that forwards to a private `string` overload — duplicated 9× across the codebase.
- **The agent-turn-killing exception:** any tool exception (HTTP timeout, validation error, NRE) bubbling all the way to `FunctionInvokingChatClient` and ending the entire agent run.

If any of these apply, you can delete the workaround code and go back to natural tool signatures after upgrading.

## What changed in 0.0.2-alpha.52

Three independent fixes land together:

1. **Kind-tolerant argument extraction.** Generated `AIFunction` wrappers no longer call `JsonElement.GetString()` / `GetInt32()` / etc. directly. They now route every extraction through `AgentFrameworkArgumentExtractor` — a public helper that handles every `JsonValueKind` correctly. Receiving a `JsonElement` of kind `Array` for a `string` parameter no longer throws — it returns the canonical JSON text via `GetRawText()`.
2. **Cancellation propagation.** `ToolResultFunctionMiddleware` no longer swallows `OperationCanceledException` (latent bug fixed). If you weren't using the middleware before, this doesn't affect you yet.
3. **Better LLM-facing failure messages.** `ToolResult.UnhandledFailure` now includes the exception type name in the default message (no `Exception.Message` body, no stack trace) so the model can distinguish `ArgumentException` from `TimeoutException` and choose a sensible recovery.

The full behavior-change list is in the [changelog](https://github.com/ncosentino/needlr/blob/main/CHANGELOG.md). The most important entries for tool authors are reproduced below.

## Migration steps

### 1. Bump the Needlr package version

In your `Directory.Packages.props` (or wherever you pin Needlr versions):

```xml
<PackageVersion Include="NexusLabs.Needlr.AgentFramework" Version="0.0.2-alpha.52" />
<PackageVersion Include="NexusLabs.Needlr.AgentFramework.Generators" Version="0.0.2-alpha.52" />
<PackageVersion Include="NexusLabs.Needlr.AgentFramework.Workflows" Version="0.0.2-alpha.52" />
```

If you have other Needlr packages referenced (`Analyzers`, `Evaluation`, etc.), bump them too — Needlr packages release together.

Run `dotnet restore` to pull the new versions.

### 2. Restore each tool to its natural shape

For every tool that you split into a `JsonElement` public method + private `string` overload, collapse them back to a single `[AgentFunction]` method that takes `string` directly.

**Before** (the workaround you wrote):

```csharp
[AgentFunctionGroup("cold-reader-findings")]
public sealed class RecordColdReaderFindingTool(IArticlePipelineRunAccessor pipelineAccessor)
{
    [AgentFunction]
    [Description("Record all findings from your evaluation in a single call. Pass a JSON array...")]
    public RecordColdReaderFindingResult RecordFindings(
        [Description("JSON array of findings. Each object: {\"severity\":\"Warning\",...}")]
        JsonElement findings)
        => RecordFindingsCore(JsonArgumentNormalization.ToJsonString(findings));

    private RecordColdReaderFindingResult RecordFindingsCore(string findingsJson)
    {
        var findings = JsonSerializer.Deserialize<FindingDto[]>(findingsJson, …);
        // …
    }
}
```

**After** (natural shape):

```csharp
[AgentFunctionGroup("cold-reader-findings")]
public sealed class RecordColdReaderFindingTool(IArticlePipelineRunAccessor pipelineAccessor)
{
    [AgentFunction]
    [Description("Record all findings from your evaluation in a single call. Pass a JSON array...")]
    public RecordColdReaderFindingResult RecordFindings(
        [Description("JSON array of findings. Each object: {\"severity\":\"Warning\",...}")]
        string findingsJson)
    {
        var findings = JsonSerializer.Deserialize<FindingDto[]>(findingsJson, …);
        // …
    }
}
```

Apply the same collapse to every tool that has the public/private overload pair. The 9 BrandGhost-style tools that need this are typically:

- `RecordColdReaderFindingTool`
- `RecordReconciledIssuesTool`
- `RecordReviewIssuesTool`
- `SetFaqEntriesTool`
- `ReplaceDeadLinksTool`
- `BatchInsertLinesTool`
- `BatchLinkInsertionTool`
- `BatchFindReplaceTool`
- `BatchReplaceTool`

(Search for `JsonArgumentNormalization.ToJsonString` to find them.)

### 3. Delete the `JsonArgumentNormalization` helper

```bash
git rm <path>/Tools/JsonArgumentNormalization.cs
```

After the previous step there should be no remaining references. `dotnet build` will fail loudly if you missed one.

### 4. Delete the architecture test

```bash
git rm <path>/AgentFunctionJsonParameterTests.cs
```

The new generator does the right thing whether the parameter is `string` or `JsonElement`. The test was enforcing your workaround, not a real invariant.

### 5. Build and run tests

```bash
dotnet build
dotnet test
```

Existing unit tests that called the `string`-typed method directly continue to work unchanged. Existing tests that called the `JsonElement`-typed method through the wrapper need updating — but step 6 below has a stronger replacement.

### 6. Add an end-to-end wrapper test for at least one tool

This is the gap the bug report called out. Hand-written tests typically call the user method directly and never see kind-mismatch failures. Adopt the pattern from [`NexusLabs.Needlr.AgentFramework.GeneratedWrapper.Tests`](https://github.com/ncosentino/needlr/tree/main/src/NexusLabs.Needlr.AgentFramework.GeneratedWrapper.Tests) for at least one canonical tool:

```csharp
[Fact]
public async Task RecordFindings_AcceptsArrayLiteral()
{
    Assert.True(AgentFrameworkGeneratedBootstrap.TryGetAIFunctionProvider(out var provider));

    var services = new Syringe()
        .UsingReflection()
        .BuildServiceProvider();

    Assert.True(provider!.TryGetFunctions(typeof(RecordColdReaderFindingTool), services, out var fns));
    var fn = Assert.Single(fns!, f => f.Name == nameof(RecordColdReaderFindingTool.RecordFindings));

    var args = new AIFunctionArguments
    {
        ["findingsJson"] = JsonDocument.Parse("[{\"severity\":\"Warning\"}]").RootElement,
    };

    await fn.InvokeAsync(args, cancellationToken: TestContext.Current.CancellationToken);
    // …assert the tool body received the expected canonical JSON…
}
```

This test class would have caught the original production crash. Without it, every new tool that has non-trivial argument shapes is still flying blind.

### 7. (Strongly recommended) Enable `.UsingToolResultMiddleware()`

Even with the kind-tolerant extraction fix, tool exceptions from your own code (database timeouts, validation, NREs) still kill the entire agent turn unless `ToolResultFunctionMiddleware` is wired in. If your bootstrap doesn't already call it:

```csharp
serviceProvider.UsingAgentFramework()
    .AddAgentFunctionsFromGenerated(...)
    .UsingResilience()             // Innermost — retries first
    .UsingToolResultMiddleware()   // Outermost — catches what resilience couldn't recover
    .BuildAgentFactory();
```

See [Tool Result Middleware](../tool-result-middleware.md) for the full behavior breakdown, including the `OperationCanceledException`-now-propagates fix and the diagnostics-counts-as-success caveat.

### 8. Verify the original failure is gone

The fastest reproduction the bug report described:

```csharp
var fn = serviceProvider.GetRequiredService<IAIFunctionProvider>()
    .GetFunctions(typeof(RecordColdReaderFindingTool), serviceProvider).Single();

var args = new AIFunctionArguments
{
    ["findingsJson"] = JsonDocument.Parse("[]").RootElement,   // Array, not String
};

await fn.InvokeAsync(args);   // Used to throw InvalidOperationException
```

After upgrade this completes without throwing. The tool body receives `"[]"` (the canonical JSON of the empty array literal).

## What to expect from the new analyzer (NDLRMAF030)

After upgrading, the analyzer will emit Info-level diagnostics on `[AgentFunction]` `string` parameters that are named like JSON (`*Json` / `*_json`) or whose `[Description]` mentions `"JSON array"` / `"JSON object"`. You have three options per parameter:

| Option | When to pick | Tradeoff |
|---|---|---|
| **A. Ignore** | Default — the kind-tolerant generator coercion makes the string-typed shape work | LLM has to send a stringified JSON; the canonical text round-trip is invisible to the tool body |
| **B. Suppress per parameter** | You're certain you want the JSON-as-string contract for this tool | One-line `#pragma warning disable NDLRMAF030` around the parameter |
| **C. Migrate to `JsonElement`** | You want typed access to the shape inside the tool body | Tool body inspects `ValueKind` directly without re-parsing |

Most BrandGhost-style tools fit option A — your `JsonSerializer.Deserialize<T[]>(arg)` body keeps working transparently because `GetRawText()` of an array IS valid JSON. Pick option C only if you want to skip the round-trip.

The diagnostic is informational, not a defect. The build will not fail because of it.

## Bonus: other improvements you can adopt

Three additional patterns are now first-class:

### Typed DTO parameters

Instead of `string findingsJson` you can use a typed DTO and let the generator emit a full property-level schema:

```csharp
[AgentFunction]
[Description("Record findings.")]
public RecordResult RecordFindings(
    [Description("Findings to record.")] FindingsBatch batch)
{
    foreach (var f in batch.Findings)
        // …
}

public sealed class FindingsBatch
{
    [Description("Each finding's severity, location, and message.")]
    public FindingDto[] Findings { get; set; } = [];

    [Description("Optional reviewer initials.")]
    public string? Reviewer { get; set; }
}
```

Limitation: the DTO must have a parameterless constructor and mutable `{ get; set; }` properties. Records with positional primary constructors and init-only properties don't currently bind through the generator.

### `Guid` / `DateTime` / `DateTimeOffset` / `TimeSpan` parameters

These now have correct schema (`{"type":"string","format":"uuid"|"date-time"|"duration"}`) and parse from the string the LLM emits. Previously these silently became `Guid.Empty` / `DateTime.MinValue` / etc. `TimeSpan` accepts both .NET round-trip (`"01:30:00"`) and ISO 8601 (`"PT1H30M"`) formats.

### `ToolResult<T, E>` for graceful failure

For tools where you want to give the LLM a recovery hint instead of a generic error:

```csharp
[AgentFunction]
public ToolResult<RecordResult, ToolError> RecordFindings(string findingsJson)
{
    if (string.IsNullOrWhiteSpace(findingsJson))
    {
        return ToolResult.Fail<RecordResult>(
            "findingsJson was empty",
            suggestion: "Provide at least one finding before calling this tool.");
    }
    // …
    return ToolResult.Ok(result);
}
```

The LLM sees `{ "error": { "message": "...", "suggestion": "..." } }` on failure and can adjust its next turn.

## Behavior changes worth knowing about

These apply to every consumer regardless of whether you adopt the migration above:

- **`string` parameter receiving `null` / `Undefined`** — was throw, now returns `""`. Code expecting null-tolerance can rely on this.
- **`string` parameter receiving non-string kinds** — was throw, now returns `GetRawText()` (canonical JSON of the literal). Existing `JsonSerializer.Deserialize<T[]>(arg)` keeps working.
- **`bool` / numeric parameter receiving wrong kind or `null`** — still throws, but with a useful kind-detail message. The previous `Convert.ToInt32(null) → 0` silent path is gone — if any tool relied on this, it now throws explicitly.
- **`UnhandledFailure` LLM-facing message** changed from `"An unexpected error occurred. Please try again."` to `"An unexpected error occurred ({ExceptionTypeName}). Please try again."` — type names only, no message body. Pass `formatter:` to opt out.
- **Top-level DTO parameters** previously got `default(T)` (silent null) on `JsonElement` Object delivery; now populate from properties.
- **Nested-object `Guid` / `DateTime` / `TimeSpan` properties** previously got `{"type":"object"}` schema (broken — model didn't know the shape); now get `{"type":"string","format":"..."}`.
