## [0.0.2-alpha.66] - 2026-06-26

### Added

- **`NexusLabs.Needlr.Generators` — `[RegisterClosedOverImplementationsOf]`: source-generate a registration of an open generic *composition* type closed over the type argument(s) of every discovered implementation of another open generic interface, exposed as a non-generic facade.** This keeps a "compose-and-expose" pattern — a growing set of typed definitions (`IFooDefinition<TData>`) surfaced uniformly behind a facade so the rest of the system can enumerate `IEnumerable<IFoo>` without knowing any `TData` — on the source-generation path, instead of a hand-maintained per-type registration list (which silently drifts when a definition is added but its wiring line is forgotten) or runtime reflection (`Assembly.GetTypes()` + `MakeGenericType` + `ActivatorUtilities`, which is AOT/trimming hostile). Mark the reusable composition with `[RegisterClosedOverImplementationsOf(typeof(IFooDefinition<>), As = typeof(IFoo))]`; the generator discovers every concrete closed implementation of `IFooDefinition<>` and emits one `services.AddSingleton<IFoo>(sp => new FooCore<TData>(...))` per discovered `TData`, resolving each constructor dependency (including `[FromKeyedServices]` parameters) closed over the same `TData` via Roslyn `Construct` — so adding a new variant means adding one definition and nothing else. The attribute exposes `As` (the facade service type, which the composition must implement), `Lifetime` (`Singleton` by default), and `AllowMultiple = true` (a composition may expose itself over several source interfaces or facades). It is a sibling to `[OpenDecoratorFor]` and lives in the `NexusLabs.Needlr.Generators` namespace to signal that it is source-generation only. Discovery spans the current assembly and referenced plain libraries — the same set `[OpenDecoratorFor]` expands over — while referenced assemblies that carry their own `[GenerateTypeRegistry]` self-register and are not re-scanned. Compile-time analyzers validate usage: **NDLRGEN035** (source must be an open generic interface), **NDLRGEN036** (composition must be an open generic with matching arity), and **NDLRGEN037** (composition must specify and implement `As`); the generator additionally emits **NDLRGEN038** and skips a registration when a discovered type argument violates the composition's generic constraints, turning a would-be runtime absence into a build-time signal rather than emitting code that fails to compile. Constraint checking is variance-safe and never skips a valid registration: `class` / `struct` / `unmanaged` / `new()` / `notnull` and non-generic constraint types are enforced precisely, while all generic constraint types (self-referential, variant, or invariant) defer to the C# compiler. Ships with generator and analyzer unit tests, integration tests that build a real `Syringe` and resolve `IEnumerable<facade>`, a runnable demo in `src/Examples/SourceGen/AotSourceGenConsoleApp`, and `docs/composed-closed-generics.md`. (#23)

### Fixed

- **The source generator (`TypeRegistryGenerator`) now emits a minimal `TypeRegistry` for an attribute-carrying assembly that has no registerable types, instead of emitting nothing — fixing a `CS0234` build failure when another project references such an assembly.** Under solution-wide `NeedlrAutoGenerate=true`, every project receives `[assembly: GenerateTypeRegistry]`, including pure-domain / contracts / abstractions libraries made up only of `record`, `enum`, and `interface` types (none of which are auto-registered). Previously the generator's producer side emitted nothing for a type-less assembly, but the consumer side still emitted `_ = typeof(<Assembly>.Generated.TypeRegistry).Assembly;` in the referencing project's generated `NeedlrSourceGenBootstrap.g.cs` force-load block. The generator now emits a minimal registry (empty `GetInjectableTypes()` / `GetPluginTypes()` providers plus a `[ModuleInitializer]` calling the two-argument `NeedlrSourceGenBootstrap.Register`) for any assembly that declares the attribute but has nothing to register. The minimal registry depends **only** on `NexusLabs.Needlr.Generators.Attributes` (which a project must already reference to use the attribute) and not on the injection packages, so it compiles even for a project that references no Needlr injection packages; it registers no `ServiceCatalog` and applies no decorators because there is nothing to register. An assembly that does not declare `[GenerateTypeRegistry]` (e.g. one with `NeedlrAutoGenerate=false`) still emits nothing.

### Changed

- **A type-less project that carries `[GenerateTypeRegistry]` now emits a minimal `TypeRegistry` and a `[ModuleInitializer]` where it previously emitted no generated output.** This is the mechanism of the `CS0234` fix above and is required so the assembly is force-loadable by consumers; at runtime its module initializer registers empty providers (harmless). ### Changed

- **`StageTermination.Custom.ToTagValue()` now returns the bounded discriminator `"Custom"` instead of its free-form `Reason`** (`NexusLabs.Needlr.AgentFramework.Diagnostics`). The `Reason` was previously used verbatim as the `termination_cause` tag on both the `pipeline.stage.completed` metric and the `pipeline.stage` span, so any run-specific text (counts, ids, short narratives) produced unbounded OpenTelemetry / Prometheus label cardinality — exactly the footgun `ToTagValue()` exists to prevent, and the one case that violated its own "stable, low-cardinality" contract. `Custom` no longer overrides `ToTagValue()`; it inherits the default case-name discriminator, so `termination_cause` is now bounded by the ~11 `StageTermination` case names for every consumer with no code change. The free-form reason is preserved everywhere it matters for diagnostics: it remains a property on the `Custom` record, round-trips through JSON unchanged (`$kind` / `Reason` / `Properties`), and is reachable via pattern matching and `Custom.Properties`. **Behavior-changing for dashboards, queries, or recording rules** that grouped `pipeline.stage.completed` (or the `pipeline.stage` span) by `termination_cause` and relied on the verbatim `Custom.Reason` — those now see `"Custom"`; read the reason from the serialized result or `Custom.Properties` instead. Third-party `IStageTermination` implementations are unaffected — their own `ToTagValue()` still drives the tag.

### Added

- **`NexusLabs.Needlr.Logging`** — new package providing cancellation-aware source-generated logging via the `[NeedlrLoggerMessage]` attribute. Mirrors the surface of Microsoft's `[LoggerMessage]` (`EventId`, `EventName`, `Level`, `Message`, `SkipEnabledCheck`) so migrating an existing logging method is a near drop-in swap, but the generated body adds a cancellation guard: when the method's first `System.Exception`-assignable parameter is a cancellation (an `OperationCanceledException` by default) the entry is **skipped entirely** by default, because cancellation is normal control flow and logging it as a warning or error is noise. Roslyn generators cannot consume one another's output, so rather than reimplementing Microsoft's generator the Needlr generator reuses the public `LoggerMessage.Define` runtime primitive — the exact allocation-free fast path Microsoft's generator targets — for up to six message parameters, and falls back to `LoggerExtensions.Log` beyond six; there is no performance penalty for the common case. The cancellation behaviour is globally overridable at startup through the static `NeedlrCancellationLogging` policy: `Behavior` (`Skip` / `Log` / `Demote`), `DemotedLevel`, and a customizable `IsCancellationPredicate`, each with a one-time, lazily-cached default read from the `__NEEDLR_CANCELLATION_LOGGING_BEHAVIOR` and `__NEEDLR_CANCELLATION_LOGGING_DEMOTED_LEVEL` environment variables. - **`TieredProviderSelectorOptions` + `ProviderFailurePolicy` + `ProviderFailureContext`** (`NexusLabs.Needlr.AgentFramework.Providers`) — declarative failure-handling policy model for `TieredProviderSelector<TQuery, TResult>`. Replaces the hardcoded `catch (ProviderUnavailableException)` with a configurable list of `ProviderFailurePolicy(Match, SkipDuration, OnHit)` rules evaluated in order against each thrown exception (first match wins). Matching policies cause fall-through to the next provider, optionally cache the failed provider as skipped for a configurable duration in a per-instance `ConcurrentDictionary<string, DateTimeOffset>` (thread-safe, in-memory, OrdinalIgnoreCase keyed), and optionally invoke an async `OnHit` callback with a typed `ProviderFailureContext(ProviderName, Exception, SkipUntil?)`. `SkipDuration` semantics: `null` = no skip, finite `TimeSpan` = skip until `now + duration`, `TimeSpan.MaxValue` (alias `ProviderFailurePolicy.IndefiniteSkip`) = skip until process restart (clamped to `DateTimeOffset.MaxValue` so the addition cannot overflow). Unmatched exceptions re-throw raw. Cancellation is excluded from policy matching — `OperationCanceledException` propagates unchanged with no skip applied. `TieredProviderSelectorOptions.Default` is a singleton preserving the framework's historical behaviour as a single matched policy (`ProviderUnavailableException` → fall through, no skip, no callback); consumers extend it via `Default with { FailurePolicies = [.. Default.FailurePolicies, ...] }`. Motivation: every non-`ProviderUnavailableException` exception type previously required a `try`/`catch`/wrap-and-rethrow boilerplate provider, and the selector had no cross-call memory of failed providers so a broken-credentials provider was retried first on every call. ### Fixed

- **🛠️ `TieredProviderSelector` quota leak on non-`ProviderUnavailableException` exceptions.** `IQuotaGate.ReleaseAsync` was previously called only on the success path and the explicit `catch (ProviderUnavailableException)` branch — any other exception thrown by `ITieredProvider.ExecuteAsync` (e.g. `HttpRequestException`, `CopilotAuthException`, `OperationCanceledException`, generic `Exception`) silently leaked the reservation acquired via `TryReserveAsync`. Custom `IQuotaGate` implementations that count outstanding reservations would drift over time. The release call has been moved into a single `finally` block driven by a `succeeded` flag, so quota is always released regardless of: success, matched-policy fall-through, unmatched re-throw, cancellation, or `ProviderFailurePolicy.OnHit` callback throwing. Three new tests (`QuotaRelease_OnUnmatchedThrow_ReleasesQuotaWithFailedFlag`, `QuotaRelease_WhenOnHitThrows_StillReleasesQuota`, plus the preserved `QuotaRelease_OnMatchedPolicy_ReleasesQuotaWithFailedFlag`) lock down the contract for every release path.
- **🛠️ Cached and reasoning token counts dropped from MEAI `UsageDetails`.**`DiagnosticsChatClientMiddleware` was reading `usage?.AdditionalCounts?.GetValueOrDefault("CachedInputTokens" / "ReasoningTokens")` — dictionary keys that no MEAI provider populates — instead of the first-class `UsageDetails.CachedInputTokenCount` and `UsageDetails.ReasoningTokenCount` properties available since MEAI 10.5.0 (already the pinned version). Result: every `ChatCompletionDiagnostics.Tokens.CachedInputTokens` was structurally `0` on every provider, blocking all observability of provider-side prompt caching — notably Azure OpenAI in production where prompt caching is reported but Needlr was discarding the values. Both the non-streaming and streaming paths in `DiagnosticsChatClientMiddleware` now read the first-class properties with `AdditionalCounts` fallback for backward compatibility with custom `IChatClient` implementations that may have populated the legacy keys. `IterativeAgentLoop` per-iteration token aggregation also previously hardcoded `0` for cached and reasoning in all three normal-completion sites (`MaxToolCallsReached`, `CompletedEarlyAfterToolCall`, end-of-iteration); now correctly aggregates from `UsageDetails` per call. `gen_ai.usage.cached_input_tokens` and `gen_ai.usage.reasoning_tokens` OTel activity tags added on per-call success spans (`DiagnosticsChatClientMiddleware`) and aggregate run spans (`DiagnosticsAgentRunMiddleware`). Motivation: a 1,571-call corpus against Azure OpenAI `gpt-4.1-2025-04-14` reported `CachedInputTokens=0` for every call.
- **🛠️ Streaming failure path discarded partial usage tokens.** When a provider emitted a `UsageContent` chunk before a streaming chat completion failed (e.g., late network drop after the model had already reported token usage), Needlr was throwing the partial usage data away. Root cause: `DiagnosticsChatClientMiddleware.HandleStreamingAsync` hardcoded `Tokens: new TokenUsage(0, 0, 0, 0, 0)` and `InputMessageCount: 0` in the failure branch even though MEAI's `buffered.ToChatResponse()` correctly aggregates any `UsageContent` chunks observed before the failure into `aggregated.Usage`. The partial `Response` itself was already being preserved — only the token counts and request size were being zeroed. Now derives `TokenUsage` from `aggregated.Usage` using the same first-class + `AdditionalCounts` mapping as the success path, and sets `InputMessageCount` from the actual message list count.
- **🛠️ Partial-iteration records lost LLM token data on exception/cancellation.** When an `IterativeAgentLoop` iteration was interrupted by an exception, an HTTP timeout, or genuine cancellation **after** one or more LLM calls had already succeeded, `RecordPartialIteration` was hardcoding `TokenUsage(0, 0, 0, 0, 0)` and `LlmCallCount: 0` in the recorded `IterationRecord`. The token data was already accumulated in loop-local variables but unreachable from the outer catch handlers. Per-iteration token counters and `llmCallCount` are now promoted to outer-scope variables (`currentIterationInputTokens`, `currentIterationOutputTokens`, `currentIterationTotalTokens`, `currentIterationCachedInputTokens`, `currentIterationReasoningTokens`, `currentIterationLlmCallCount`), reset at the start of each iteration alongside the existing `currentIterationIndex` / `currentIterationToolCalls` / `currentIterationStopwatch` state. `RecordPartialIteration` extended to accept the five token counters and `llmCallCount`; passes them through to the partial `IterationRecord` instead of zeros. All three call sites updated (cancellation, HTTP-timeout / non-user `OperationCanceledException`, generic `Exception`). Same class of bug as the streaming failure path above: telemetry data was thrown away on the unhappy path even though it had been observed and accumulated.
- **🛠️ `InMemoryWorkspace` path aliasing** — the previous `NormalizePath` was just `path.Replace('\\', '/').TrimStart('/')`, so `kb/foo.md`, `./kb/foo.md`, `kb//foo.md`, `kb/./foo.md`, `/kb/foo.md`, and `kb/foo.md/` all produced **distinct dictionary keys** despite referring to the same logical file. Two agents writing the same logical file under different aliases would silently produce two entries and downstream readers would see "missing file" gaps. Now all six variants collide on the canonical `kb/foo.md` key, matching the (now-explicit) `IWorkspace` contract. Filed against `0.0.2-alpha-0054` and observed as far back as `0.0.2-alpha-0049`. Motivation: observed during a knowledge-base pipeline build (2026-05-06).
